#Issue 6: Boost Force Direction -- "Linearly Interpolated from 0 to Strength"

Severity: MAJOR Location: Section 4.3 Boost/Turbo Force Application Claim: "The boost force is linearly interpolated: from 0 at start to strength * modelScale at end of duration" Evidence Check: The decompiled code at NSceneVehiclePhy__ComputeForces.c line 105-106 shows:

local_b8 = ((float)(param_5 - uVar2) / (float)*(uint *)(lVar6 + 0x16e0)) *
           *(float *)(lVar6 + 0x16e4) * *(float *)(*(longlong *)(lVar6 + 0x1bb0) + 0xe0);

The formula is (elapsed / duration) strength modelScale. At elapsed=0, force=0. At elapsed=duration, force=strengthmodelScale. This IS a linear ramp from 0 to max. However, the document says "from 0 at start to strength modelScale at end" -- this implies the boost INCREASES over time, reaching maximum at the END. For a turbo/boost, this is counterintuitive (most games apply boost strongest at the beginning). The code is accurately described but the interpretation deserves scrutiny: this means the car accelerates more as the boost is about to expire. Verdict: CORRECT in code description, but the reviewer notes this is an unusual boost curve (ramping up, not decaying). The document's own section 6.3 says "force(t) = (t / duration) strength modelScale" which matches. However, section 6.4 in the TMNF crossref (doc 14) says "Decay: linear over duration" which contradicts this -- in TMNF the force decays, in TM2020 the decompiled code shows it ramps UP. Recommendation: Add explicit commentary that the TM2020 boost force INCREASES linearly, not decreases. This is a significant difference from TMNF's documented behavior and is critical for recreation.

#Issue 12: Fragile Check Condition 3 -- "nibble values 4+ only"

Severity: MINOR Location: Section 6.4 Claim: "Status nibble > 3: The vehicle status (nibble - 2) must be greater than 1, meaning nibble values 4+ only" Evidence Check: Code shows (1 < ((uint )(lVar9 + 0x128c) & 0xf) - 2). For nibble=4: (4-2)=2, 1<2 is true. For nibble=3: (3-2)=1, 1<1 is false. For nibble=2: (2-2)=0, 1<0 is false. For nibble=1: (1-2)=0xFFFFFFFF (unsigned wrap), 1<0xFFFFFFFF is true (unsigned comparison). Wait -- this is unsigned arithmetic. The code does ((uint )(lVar9 + 0x128c) & 0xf) - 2 which is unsigned subtraction. For nibble=0: 0-2 = 0xFFFFFFFE, and 1 < 0xFFFFFFFE is TRUE. For nibble=1: 1-2 = 0xFFFFFFFF, and 1 < 0xFFFFFFFF is TRUE. So nibble values 0, 1, and 4+ all pass. Only nibble values 2 and 3 fail. Verdict: WRONG -- the document claims "nibble values 4+ only" but unsigned underflow means nibble values 0 and 1 ALSO pass the check Recommendation: Correct to: "nibble values 0, 1, and 4+ pass the check. Only nibble values 2 and 3 are excluded." This is significant because it means the fragile check can trigger in more states than documented.
